Flume - 日志采集

日志采集

各种软件系统形态各异，操作系统存在差异，Un*x、Linux、Window、MacOS、AndroidOS等等。服务器软件也部署在各种不同的服务器中。 这种时候需要收集各系统的日志，可以通过监听日志文件的变化，将变化数据以消息的模式输送到目标日志系统。这样做可以最大限度的降低 对原有系统的侵入性。

Java 文件监控

Apache Common-IO

Apache Common IO 库提供一个文件监控的API，使用方便，其采用了一种定时轮询的方式，检查目录和文件的变化以达到监听的目的。 这种方式造成的困扰是CPU的时间消耗巨大，如果文件数量特别多的情况下，情况会更糟糕。如果需要监听的文件数目很少，例如 几个文件，并且 文件本身又十分巨大的话，那还是值得一试。

JDK WatchService

比较新版本的JDK提供一个更加有效的文件变化监听工具 WatchService, JDK 使用Native方法实现更加底层的阻塞事件监听，因此使用时感受 是对CPU的消耗没有那么大。

Flume

Flume是Cloudera提供的一个高可用的，高可靠的，分布式的海量日志采集、聚合和传输的系统，Flume支持在日志系统中定制各类数据发送方， 用于收集数据；同时，Flume提供对数据进行处理，并写到各种数据接受方（可定制）的能力。主要有三层抽象 Flume Sources、Flume Channel、Flume Sink，通过各种组合可以完成多种处理流程。

Flume 的流程处理都是通过配置文件的形势组织数据流转过程，流转过程中的数据处理的具体实现，还提供定制扩展功能。

Spooling Directory Source

Spooling Directory Source的目的就是监听磁盘文件，将变化的数据通过Flume流转传送出去，后续只需要使用合适的Channel和Sink就可以完成 一个完整的数据采集传输工作了。

Flume的流转过程，Source的类型，Channel的类型，Sink的类型都是通过配置文件来管理的。

因为真实数据系统的多样性，Flume 提供丰富的 Source 、Channel、Sink 类型，可以轻松覆盖大多数的应用场景。 即使如此，任然存在大量的定制需求。

这里有详细的开发指南

Client

Client在 Event的起Source点进行操作并将其交付给Flume代理。客户端通常在他们正在使用数据的应用程序的进程空间中运行。 Flume目前支持Avro，log4j，syslog和Http POST（带有JSON主体）作为从外部Source传输数据的方法。此外， 还有一个ExecSource可以使用本地进程的输出作为Flume的输入。很可能有一个用例，这些现有的选项是不够的。在这种情况下， 您可以构建自定义机制以将数据发送到Flume。有两种方法可以实现这一目标。第一个选项是创建一个自定义客户端， 与Flume现的Source之一（如 AvroSource或SyslogTcpSource）进行通信。客户端应将其数据转换为这些Flume Source所理解的消息。 另一种选择是编写一个自定义Flume Source，它使用某种IPC或RPC协议直接与您现有的客户端应用程序进行通信， 然后将客户端数据转换为Flume Event，以便向下游发送。请注意，所有 Event都存储在Flume代理的渠道必须作为Flume Event存在。

Client SDK

虽然Flume包含许多内置机制（即Source）来摄取数据，但通常需要能够直接从自定义应用程序与Flume进行通信。 Flume Client SDK是一个库，使应用程序能够连接到Flume并通过RPC将数据发送到Flume的数据流。

RPC客户端接口

Flume的RpcClient接口的实现封装了Flume支持的RPC机制。用户的应用程序可以简单地调用Flume Client SDK的append（Event） 或appendBatch（List ）来发送数据，而不用担心底层的消息交换细节。用户可以通过直接实现Event接口， 使用简单实现（如SimpleEvent类）或使用 EventBuilder重载的withBody（）静态帮助器方法来提供所需的Event arg 。

LoadBalancing RPC 客户端

Flume Client SDK还支持RpcClient，该RpcClient可在多个主机之间进行负载平衡。这种类型的客户端采用以空格分隔的：列表， 这些列表代表组成负载平衡组的Flume代理。可以使用负载均衡策略配置此客户端，该策略可以随机选择已配置的主机之一，也可以以循环方式选择主机。 您也可以指定自己的自定义类，该类实现LoadBalancingRpcClient $ HostSelector接口，以便使用自定义选择顺序。在那种情况下， 需要将定制类的FQCN指定为host-selector属性的值。 LoadBalancing RPC客户端当前不支持节俭。 如果启用了退避，则客户端将暂时将出现故 障的主机列入黑名单，从而导致这些主机被排除在选定的故障转移主机之外，直到给定超时为止。超时结束后，如果主机仍然无响应，则这被认为是顺 序故障，超时将成倍增加，以避免潜在地卡在无响应主机上的长时间等待中。 可以通过设置maxBackoff（以毫秒为单位）来配置最大退避时间。 maxBackoff的默认值为30秒（在OrderSelector类中指定，该类是两种负载平衡策略的超类）。随着每个顺序故障，退避超时将成倍增加，直到最大 可能的退避超时。最大可能的回退限制为65536秒（约18.2小时）。
